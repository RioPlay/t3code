== Purpose and scope

This strategy defines how T3 Code Mobile achieves *verifiable Android parity* without
regressing iOS. It closes known gaps:

* No `.maestro/` directory or Maestro CI job
* No `capabilities.test.ts`
* `remoteRegistration` tests mock `Platform.OS === "ios"` only
* `npx expo-doctor` not enforced in CI
* No mobile crash reporting
* `smoke-agent-push` requires relay + FCM test harness
* Sparse `testID` coverage blocks reliable Maestro selectors

*In scope:* `apps/mobile`, native modules under `apps/mobile/modules/`, relay device
registration contract, CI workflow changes, manual smoke ownership.

*Out of scope:* Store release marketing, full Pierre RN parity, Kotlin review diff unless
REV-007 perf gate fails.

=== Related requirement IDs

Implementation requirements remain in link:requirements.adoc[requirements.adoc].
This document adds xref:qa-requirements[QA-*] and xref:obs-requirements[OBS-*] IDs for
verification and observability work.

== Test pyramid

The pyramid optimizes for *fast feedback at the base* and *agent/review confidence at the top*.
Every layer maps to milestone gates (xref:release-gates[Release gates]).

[source]
----
                    ┌─────────────────────┐
                    │  Manual smoke       │  smoke-matrix.adoc
                    │  (tablet, OEM push) │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Maestro E2E        │  .maestro/smoke-*.yaml
                    │  (5 flows, Android) │
                    └──────────┬──────────┘
                               │
          ┌────────────────────▼────────────────────┐
          │  Integration (Vitest + mocked native)   │
          │  remoteRegistration, review VM, relay   │
          └────────────────────┬────────────────────┘
                               │
          ┌────────────────────▼────────────────────┐
          │  Unit (Vitest, hermetic)                │
          │  capabilities, parsers, tokens, nav     │
          └─────────────────────────────────────────┘
----

=== Layer 1: Unit tests (target: < 30 s in CI)

* *Runner:* `vp test` / `vp run test` (existing `Test` job in `.github/workflows/ci.yml`)
* *Location:* colocated `*.test.ts` / `*.test.tsx` under `apps/mobile/src/`
* *Policy:* New Android-parity code ships with tests; no `Platform.OS` branches without both
  platform fixtures (see xref:android-unit-requirements[Android unit test requirements]).

*Primary coverage:*

* `platform/capabilities.ts` resolution and `FORCE_*` overrides (CAP-003)
* Review JS list row rendering helpers, collapse state, highlight engine selection (REV-001–004)
* Composer token collection and strip-mode payload (CMP-001, CMP-002)
* Agent registration payload shape and channel ID constants (AGT-001, AGT-002, AGT-006)
* Navigation deep-link parsers used by `useAgentNotificationNavigation` (AGT-004)

*Anti-patterns:*

* Asserting on full `ReviewSheet` pixel output
* Live network calls to relay in unit tests
* iOS-only `Platform` mocks without an Android counterpart

=== Layer 2: Integration tests (target: < 60 s additional in CI)

* *Runner:* same Vitest job; tests may use `ManagedRelay` test layer (pattern in
  `remoteRegistration.test.ts`)
* *Scope:* Multi-module flows with mocked `expo-notifications`, `expo-constants`, SecureStore

*Required integration suites:*

[cols="2,2,1",options="header"]
|===
| Suite | Validates | Milestone

| `remoteRegistration.test.ts` (Android path)
| FCM token accepted, relay POST, iOS Live Activity skipped
| M1

| `registrationPayload.test.ts`
| `RelayDeviceRegistrationRequest` for `type: android`
| M1

| Review view-model alignment (M2)
| Shared cache keys match web `diffRendering.ts`
| M2

| Composer send pipeline
| Strip/mirror tokens reach outbox unchanged
| M1/M2
|===

=== Layer 3: Maestro E2E (target: nightly + native-impacting PRs)

* *Tool:* https://maestro.mobile.dev/[Maestro] on Android emulator (API 34 phone, API 34 tablet)
* *Location:* `apps/mobile/.maestro/` (repo root path: `apps/mobile/.maestro/`)
* *Flows:* five smoke flows per QA-004; see xref:maestro-flows[Per-flow Maestro specifications]
* *Runtime:* Development dev-client build with xref:maestro-auth-bypass[auth bypass] fixture

*When Maestro runs:* see xref:ci-workflow[CI workflow changes]. Not on every PR until M1
testID contract is complete (QA-006).

=== Layer 4: Manual smoke (milestone sign-off)

* *Artifact:* link:smoke-matrix.adoc[smoke-matrix.adoc]
* *Owners:* see xref:smoke-ownership[Smoke matrix ownership process]
* *When:* Required for M1/M2/M3 milestone sign-off; spot-check on M4 optional features
* *Devices:* Minimum matrix — Pixel 8 (phone), Pixel Tablet or foldable inner screen (M2+),
  one Samsung device for OEM notification variance (M1 agent push)

*Manual-only scenarios (not automated in M1–M2):*

* Clerk production sign-in on physical device
* Bluetooth keyboard shortcuts (M3)
* Ghostty NDK terminal fidelity (M3)
* Dark mode visual pass across sheets
* OEM battery optimization blocking FCM

== CI workflow changes

Current `.github/workflows/ci.yml` runs `check`, `test`, `mobile_native_static_analysis`, and
`release_smoke`. Android parity adds *conditional* jobs without slowing unrelated PRs.

=== Job matrix (proposed)

[cols="2,2,2,2",options="header"]
|===
| Job | Runs on | Trigger | Command / action

| `check` (existing)
| ubuntu
| every PR + `main`
| `vp check`, `vpr typecheck`

| `test` (existing)
| ubuntu
| every PR + `main`
| `vp run test` (includes new mobile unit tests)

| `mobile_native_static_analysis` (existing)
| macos
| every PR + `main`
| `vp run lint:mobile`

| `mobile_expo_doctor` (new) — QA-005
| ubuntu
| PRs touching `apps/mobile/**`, `app.config.ts`, `modules/**`, root `package.json` mobile deps
| `cd apps/mobile && npx expo-doctor`

| `mobile_capabilities_gate` (new) — QA-007
| ubuntu
| same paths as expo-doctor
| `vp test run apps/mobile/src/platform/capabilities.test.ts`

| `mobile_maestro_android` (new) — QA-004
| ubuntu + KVM
| `main` nightly; PRs with label `qa:maestro`; native-impacting PRs after M1
| Maestro flows on API 34 emulator

| `mobile_fingerprint_review` (new) — QA-008
| ubuntu
| PRs changing `app.config.ts`, `apps/mobile/modules/**`, `eas.json`
| Expo fingerprint diff comment (reuse preview workflow logic)

| `mobile_ios_regression_unit` (new) — QA-003
| ubuntu
| every PR touching `apps/mobile/src/**`
| Vitest with `EXPO_PUBLIC_FORCE_JS_REVIEW=1` and `EXPO_PUBLIC_FORCE_NITRO_MARKDOWN=1`

| `mobile_ios_simulator_weekly` (optional, deprioritized) — QA-009
| macos
| Weekly cron only
| iOS simulator smoke *only if* unit regression job cannot cover changed surface
|===

== testID contract (QA-006)

Maestro selectors *must* use `id:` (testID) for stability. Text and accessibility labels are
*secondary* fallbacks only.

=== Naming rules

* *Format:* `{surface}-{element}[-{qualifier}]`
* *Case:* kebab-case, lowercase
* *Stability:* Never encode dynamic UUIDs in testID; use accessibility label for thread title
  search, or tap by list index only in fixtures with seeded data
* *Platform:* Same testID on iOS and Android for cross-platform Maestro reuse
* *Tier branching:* When JS review replaces native, use `review-js-diff-list` not
  `review-native-diff-view`

== Per-flow Maestro specifications

=== Shared: auth bypass [[maestro-auth-bypass]]

Maestro cannot complete Clerk OAuth interactively in CI. Use a *development-only* bypass:

* *Mechanism:* `EXPO_PUBLIC_MAESTRO_AUTH_BYPASS=1` + secure dev fixture that seeds
  `SavedRemoteConnection` and mock relay token provider
* *Gate:* Compile-time strip in production builds (`app.config.ts` extra guard)
* *Subflow:* `fixtures/auth-bypass.yaml` — launch app, wait for `home-screen`

[source,yaml]
----
# fixtures/auth-bypass.yaml
- runFlow:
    when:
      visible:
        id: "home-screen"
    commands:
      - assertVisible:
          id: "home-screen"
----

If bypass not ready in week 1, use *pre-authenticated dev client* installed on emulator with
persistent SecureStore (document in `apps/mobile/README.md`).

=== `smoke-launch.yaml` (M1)

* *Covers:* QA-004, smoke-matrix Build & bootstrap (partial)
* *Preconditions:* Auth bypass active; at least one environment paired in fixture
* *Steps:*

[source,yaml]
----
appId: ${APP_ID}
---
- launchApp
- runFlow: fixtures/auth-bypass.yaml
- assertVisible:
    id: "home-screen"
- assertVisible:
    id: "home-thread-list"
- scroll:
    id: "home-thread-list"
    direction: DOWN
- stopApp
----

* *Failure triage:* Splash hang → Metro not running; blank home → workspace atom hydration

=== `smoke-review.yaml` (M1)

* *Covers:* REV-001–003, NAV routes ThreadReview
* *Fixture:* Thread with multi-file diff (seed via relay or bundled mock state)
* *Selectors:* `home-thread-row` → `thread-screen` → `thread-accessory-review` →
  `review-sheet` → `review-js-diff-list`

[source,yaml]
----
- runFlow: fixtures/auth-bypass.yaml
- tapOn:
    id: "home-thread-row"
    index: 0
- assertVisible:
    id: "thread-screen"
- tapOn:
    id: "thread-accessory-review"
- assertVisible:
    id: "review-sheet"
- assertVisible:
    id: "review-js-diff-list"
- scroll:
    id: "review-js-diff-list"
    direction: DOWN
- tapOn:
    id: "review-diff-file-row"
    index: 0
- back
- assertVisible:
    id: "thread-screen"
----

* *Assertions:* No crash = pass; native diff view must *not* appear on Android

=== `smoke-agent-push.yaml` (M1) — QA-010

* *Covers:* AGT-001–004
* *Dependency:* Relay test endpoint or `scripts/seed-relay-push.sh` sends FCM data message
  with deep link to seeded thread
* *Cannot run* until `remoteRegistration.ts` accepts Android tokens (AGT-001)

*CI architecture:*

[source]
----
relay (staging) ──FCM──► emulator device
       ▲
       └── Maestro seed script (device registration id from logs)
----

* *Steps:*

[source,yaml]
----
- runFlow: fixtures/auth-bypass.yaml
- runScript: scripts/seed-relay-push.sh
- pressKey: Home
- tapOn:
    text: "Waiting for approval"   # notification shade; OEM-specific fallback
- assertVisible:
    id: "thread-screen"
----

* *Flake note:* Notification shade UI varies by OEM; prefer relay-triggered *in-app* deep link
  assertion as secondary path (QA-010)

== iOS regression strategy

Goal: *shared Tier 0 code must not rot iOS Tier 2 paths* while preferring fast unit tests over
expensive simulator runs (QA-003, QA-009).

=== Primary: FORCE_* unit regression (every PR)

Run in `mobile_ios_regression_unit` job:

[source,bash]
----
EXPO_PUBLIC_FORCE_JS_REVIEW=1 \
EXPO_PUBLIC_FORCE_NITRO_MARKDOWN=1 \
vp test run apps/mobile/src/features/review apps/mobile/src/platform
----

*Assertions:*

* Native code paths not invoked when flags set (mock resolvers)
* iOS default *without* flags still selects native when mocks return views (snapshot of
  `capabilities.review.native`)

=== Secondary: iOS dogfood in development

Document in `.env.example`:

[source,bash]
----
EXPO_PUBLIC_FORCE_JS_REVIEW=1 vp run ios:dev
EXPO_PUBLIC_FORCE_NITRO_MARKDOWN=1 vp run ios:dev
----

Engineers dogfood Tier 0 on iOS weekly during M1–M2.

=== Tertiary: Weekly simulator (deprioritized) — QA-009

* *When:* Only for changes to native Swift modules (`apps/mobile/modules/**/*.swift`) that
  unit tests cannot exercise
* *Not required for:* Pure JS Tier 0 changes, Android-only Kotlin, relay contract changes
* *Command:* `vp run ios:dev` + manual smoke-matrix rows marked iOS Tier 2

== Observability

Mobile currently has *no crash reporting* (OBS-001). Agent push has no delivery funnel metrics.
Perf baselines are undocumented.

=== OBS-001: Crash reporting (M1)

* *Requirement:* Integrate Sentry (or Expo Application Services crash reporting) for
  `development`, `preview`, and `production` variants
* *PII policy:* Scrub relay tokens, Clerk session, diff content bodies
* *Breadcrumbs:* Navigation route changes, agent phase transitions, registration errors
* *Release mapping:* EAS build id + git SHA in `release` field
* *Gate:* OBS-001 blocking for M2 sign-off (T1 tier)

=== OBS-002: Notification delivery funnel (M1)

Track client-side events (analytics or structured logs → relay OTLP):

[cols="2,3",options="header"]
|===
| Event | Properties

| `agent_push.permission_requested`
| platform, granted/denied

| `agent_push.token_acquired`
| platform, token_type (apns/fcm)

| `agent_push.registered_relay`
| environment_id, success, error_code

| `agent_push.received`
| phase, thread_id (hashed)

| `agent_push.opened`
| cold_start boolean, deep_link_success

| `agent_push.registration_skipped`
| reason (ios_gate_removed, permissions_denied)
|===

*Dashboard:* Staging funnel — registration → receive → open must be >90% in harness tests

== Dev onboarding: Android TTHW checklist

*TTHW* = Time To Hello World. Target: *< 90 minutes* for a new engineer on Linux/macOS with
Android Studio installed.

=== Before you start

* [ ] Clone repo; `vp install` at root
* [ ] Copy `.env.example` → `.env.local`; set `T3CODE_CLERK_PUBLISHABLE_KEY` (dev instance)
* [ ] Install Android Studio: SDK 34, NDK per `app.config.ts`, accept licenses
* [ ] Create AVD: Pixel 8 API 34; enable hardware acceleration (KVM on Linux)
* [ ] Install Maestro CLI
* [ ] Read link:plan.adoc[plan.adoc] Executive summary + Phase 1

=== Build and run (first success)

[source,bash]
----
# Repo root
vp install

# Start Metro (terminal 1)
cd apps/mobile && vp run dev:client

# Build + install dev client (terminal 2) — first run ~15–25 min
cd apps/mobile && vp run android:dev
----

* [ ] App launches to Clerk sign-in or home
* [ ] `npx expo-doctor` exits 0 from `apps/mobile`
* [ ] `vp test run apps/mobile` passes

=== Pair environment

* [ ] Sign in via Clerk
* [ ] Pair remote environment (QR or manual) — same as iOS flow
* [ ] Home shows at least one thread

=== Verify Tier 0 review (Android)

* [ ] Open thread → Review → multi-file diff scrolls without crash
* [ ] Confirm JS list (`review-js-diff-list` once testID lands), not native view

=== Verify agent registration (after AGT-001 lands)

* [ ] Enable notifications on emulator (extended controls)
* [ ] Settings → Connections shows registered device
* [ ] Logcat contains successful relay registration (no iOS-only skip)

=== Run Maestro smoke-launch

[source,bash]
----
maestro test apps/mobile/.maestro/flows/smoke-launch.yaml
----

* [ ] Flow green with auth bypass or pre-seeded session

=== Common blockers

[cols="2,3",options="header"]
|===
| Symptom | Fix

| `prebuild` fails NDK version
| Match `expo-build-properties` in `app.config.ts`

| Metro connection refused on emulator
| `adb reverse tcp:8081 tcp:8081`

| Clerk redirect fails
| Check scheme `t3code-dev` in `app.config.ts`

| Review crashes on open
| Expected pre-M1; verify `capabilities.review.native` is false

| FCM token null on emulator
| Use Google Play image; sign into Google account on AVD
|===

=== Escalation

* #mobile-dev channel; attach `npx expo-doctor` output + logcat snippet
* File issue with requirement ID prefix (e.g. `CAP-002: Review crash on Android`)
