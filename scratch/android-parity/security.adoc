== Purpose

Security review of the Android parity plan before implementation. Identifies threat models,
PII boundaries, token-handling requirements, and milestone acceptance criteria for M0–M2.

*Scope:* FCM push delivery, notification deep links, relay device registration, DPoP auth,
WebView terminal interim (M2), ongoing agent notifications (M2).

*Out of scope:* Ghostty NDK (M3), optional FGS (M4), Kotlin review diff, store compliance audits.

---

== Executive summary

[cols="2,3",options="header"]
|===
| Area | Assessment

| Relay registration auth
| *Strong.* `POST /v1/mobile/devices` requires DPoP-bound `mobile:registration` scope;
  unchanged for Android. Primary gap is *client-side iOS gate*, not missing server controls.

| Push payload PII
| *Adequate with gaps.* Relay `notificationForAggregate()` limits visible text to thread title +
  status; no prompt/file content. Gaps: lock-screen exposure of titles, `environmentId`/`threadId`
  in FCM `data`, device `label` may leak user-assigned names to relay logs.

| Deep link handling
| *Strong client validation.* `notificationPayload.ts` allowlists `/threads/{env}/{thread}` only.
  Gap: deep link does not cryptographically bind to recipient; app must enforce environment ACL on
  navigation.

| Token storage (mobile)
| *Mostly strong.* DPoP proof key and relay access tokens in `expo-secure-store`. FCM token not
  persisted locally (obtained at registration). Gap: FCM token stored plaintext in relay DB
  (inherited iOS pattern).

| WebView terminal (M2)
| *Not yet implemented — high design risk.* Bundled xterm.js is the safe path; any remote URL,
  `originWhitelist={["*"]}`, or permissive `postMessage` bridge would introduce XSS and shell
  escape vectors. Contrast: `WorkspaceFileWebPreview.tsx` already uses `originWhitelist={["*"]}`
  for file preview — terminal must *not* copy that pattern.

| FCM credentials (relay)
| *Elevated blast radius.* Firebase Admin service account exceeds APNs key scope; requires
  least-privilege IAM, secret rotation, and log redaction (REL-041).
|===

== Threat model

=== Assets

[cols="2,3",options="header"]
|===
| Asset | Sensitivity

| Clerk session / OAuth bearer
| High — cloud account identity; used only for DPoP token exchange, not sent to FCM.

| DPoP proof private key (`t3code.cloud.dpop-proof-key`)
| High — binds relay access tokens to device; stored in SecureStore.

| Relay DPoP access token (`mobile:registration`, `environment:connect`)
| Medium-high — short-lived; scoped; cached in SecureStore per account/thumbprint.

| FCM registration token (`pushToken` in relay DB)
| Medium — allows push to device for linked user; plaintext in Postgres.

| FCM service account (relay worker)
| Critical — can send push to all registered Android devices in project.

| Agent notification payload (`title`, `body`, `data`)
| Medium — thread titles and status on lock screen; identifiers in `data`.

| Environment credential (relay-issued)
| High — unchanged; environment-authenticated publish path.

| Terminal I/O stream (WebView, M2)
| High — bidirectional shell channel; malicious escape sequences or bridge abuse → code execution
  in WebView context.
|===

*Assumptions:* Relay and environment signing keys are not compromised. Google FCM infrastructure
is trusted for delivery confidentiality in transit, not for hiding notification content from the
device owner or Google.

=== FCM push — threats and mitigations

[cols="2,2,3",options="header"]
|===
| Threat | Actor | Mitigation (existing / required)

| Forged device registration
| Remote attacker without Clerk session
| DPoP `mobile:registration` + Clerk subject token (REL-040, SEC-010). *Required:* remove iOS-only
  client gate so Android actually uses this path (AGT-001).

| Cross-user token binding
| Attacker registers victim's stolen FCM token
| Relay `Devices.register` claim-before-upsert on `push_token` unique index (SEC-011). Permanent
  FCM failure clears token (REL-026).

| Unauthorized push send
| Attacker calls FCM API directly
| FCM service account in worker secrets only; not in mobile app (SEC-020, REL-041).

| Relay DB breach exposes push tokens
| Insider / SQL injection
| Inherited risk — tokens plaintext in `relay_mobile_devices.push_token`. *Required:* DB access
  controls, encryption at rest (infra), token invalidation on logout (SEC-012). No new Android-
  specific regression.

| Malicious environment publishes misleading notifications
| Compromised or malicious linked environment
| Publish requires environment credential JWT + signed proof; relay sanitizes payload (SEC-030).
  User links environment explicitly — social-engineering risk remains.

| Notification content overshare on lock screen
| Shoulder surfing, backup of notification shade
| Payload limited to title/status (SEC-031); no prompt text. *M2:* ongoing notification must not
  add body fields (SEC-032).

| FCM `data` payload inspection
| Malware on device, backup tools
| `data` carries `environmentId`, `threadId`, `deepLink` only — no message content (SEC-031).
  Treat as opaque identifiers, not secrets.

| OEM / battery kill drops push
| Platform
| Availability risk, not confidentiality. In-app polling fallback documented in plan — must not
  bypass permission preferences (SEC-033).
|===

==== Push delivery path (Android parity)

Mirror `ApnsDeliveries.notificationForAggregate()` → `FcmDeliveries` with shared sanitizer
(`agentActivityPayloads.ts`). Android skips all `live_activity_*` kinds (REL-021).

[source]
----
Environment publish (JWT)
  → AgentActivityPublisher.publishForDeliveryUser
  → MobileDeliveries.sendForTarget (platform dispatch)
  → FcmDeliveries.sendPushNotification (Android)
  → FCM HTTP v1 (signed queue job, REL-029)
  → expo-notifications on device
----

Queue jobs must verify HMAC before send (mirror `ApnsDeliveries.processSignedJob`). Stale token
check (`isCurrentSignedJobToken`) prevents delivery after unregister/rotation.

=== Deep links — threats and mitigations

Notifications arrive through an untrusted channel (FCM). Treat all `data` fields as attacker-
controlled even though relay is the nominal sender.

[cols="2,3",options="header"]
|===
| Threat | Mitigation

| Open redirect / external URL
| Client `normalizeThreadDeepLink()` rejects `https://`, `//`, `?`, `#`, non-`/threads/` paths
  (`notificationPayload.ts` tests). Relay `sanitizeDeepLink()` blocks `//` and truncates to 512
  chars.

| Path traversal in thread IDs
| Client re-encodes via `encodeURIComponent` after decode; fixed 4-segment path shape.

| Deep link to unauthorized environment
| *Gap — app-layer.* Navigation via `useLinkTo` does not validate user owns environment. *Required:*
  thread route must fail closed if environment not in linked set (SEC-040).

| Duplicate navigation on cold start
| `routeAgentNotificationResponseOnce` dedupes by notification identifier — good.

| Phishing via notification title/body
| Relay controls copy from sanitized aggregate; environment cannot inject arbitrary FCM fields if
  sanitizer enforced. User sees thread title — low-grade phishing risk acceptable for product.
|===

*FCM vs APNs shape:* FCM delivers the same logical fields in `data` (`environmentId`, `threadId`,
`deepLink`). Mobile parser is transport-agnostic — SEC-041 requires FCM staging tests reuse
`notificationNavigation.test.ts` cases.

=== WebView terminal (M2) — threats and mitigations

Planned: bundled `@xterm/xterm` static assets, `postMessage` bridge `{ data }`, `{ cols, rows }`
(plan §2.6). This is a *new attack surface* on Android.

[cols="2,2,3",options="header"]
|===
| Threat | Severity | Mitigation (required before ship)

| XSS via remote HTML/JS in WebView
| Critical
| *SEC-050:* Load terminal from `file://` or inline bundled HTML only. `originWhitelist` must be
  `["file://"]` or absent (default deny). *Never* load remote URLs.

| `postMessage` injection from WebView content
| High
| *SEC-051:* RN bridge validates message shape (schema); ignore unknown types. WebView JS must not
  expose generic `ReactNativeWebView.postMessage` handlers beyond terminal contract.

| Malicious terminal output (ANSI/OSC hyperlinks)
| Medium
| xterm.js may render clickable links. *SEC-052:* Disable or intercept OSC 8 hyperlinks in
  mobile WebView profile; do not `linkTo` arbitrary URLs from terminal output.

| Shell escape via bridge
| High
| Bridge forwards keystrokes only as `{ data: string }` to existing RPC — no `eval`, no file URLs.
  Reuse native terminal RPC authorization (environment connection already authenticated).

| Supply-chain in xterm bundle
| Medium
| Pin `@xterm/xterm` version; `vp check` lockfile; no CDN runtime load.

| Clipboard exfiltration from terminal buffer
| Low
| Terminal buffer may contain secrets — document in UX; optional: disable long-press copy in
  WebView tier (SEC-053, optional).

| Contrast with file preview WebView
| Medium
| `WorkspaceFileWebPreview` uses `originWhitelist={["*"]}` for workspace file URIs — acceptable
  for read-only preview with separate threat model. *Do not reuse for terminal.*
|===

*Ghostty NDK (M3):* Native terminal removes WebView XSS class; WebView interim must ship with
SEC-050–052 as hard gates.

== Token storage and relay registration auth

=== Mobile local storage

[cols="2,2,2",options="header"]
|===
| Secret | Store | Key / notes

| DPoP proof key pair
| `expo-secure-store`
| `t3code.cloud.dpop-proof-key` — ES256 P-256 private JWK

| Relay DPoP access tokens
| `expo-secure-store`
| `t3code.cloud.relay-access-tokens` — JSON cache, scoped, thumbprint-bound

| Agent awareness `deviceId`
| `expo-secure-store`
| `t3code.agent-awareness.device-id` — UUID, not authentication secret

| FCM registration token
| *Not persisted locally*
| Read at registration via `getDevicePushTokenAsync()`; sent to relay in `pushToken`

| Clerk session
| Clerk SDK
| Out of scope — managed by `@clerk/clerk-expo`
|===

*SEC-001:* Sign-out path must call `unregisterDevice` + `managedRelayAccessTokenStore.clear` +
`resetTokenCache` (verify `CloudAuthProvider` Android path matches iOS).

*SEC-002:* Dev logs must not print full FCM/APNs tokens. Existing pattern: `tokenSuffix` last 8 chars
in live-activity debug only — extend to Android FCM rotation logs.

=== Relay registration authentication flow

[source]
----
1. User signs in (Clerk bearer)
2. Mobile exchanges Clerk token for DPoP access token
   POST /v1/client/dpop-token
   scope includes mobile:registration
   proof key thumbprint bound (cnf.jkt)
3. Mobile registers device
   POST /v1/mobile/devices
   Authorization: DPoP <access_token>
   DPoP: <proof JWT for this request>
   payload: RelayDeviceRegistrationRequest (Android union member)
4. Relay RelayDpopClientAuth middleware validates token + proof
5. MobileRegistrations.registerDevice persists under authenticated userId
----

*Current Android gap:* `remoteRegistration.ts` gates steps 3–4 behind
`canRegisterRemoteLiveActivities()` (`Platform.OS === "ios"`) and accepts only `token.type ===
"ios"`. This is a *functional* block, not a security control. Android parity must:

* Register push on `Platform.OS === "android"` without Live Activity prerequisites (SEC-010).
* Keep Live Activity registration iOS-only (REL-022).
* Use separate code paths: `canRegisterRemotePush()` vs `canRegisterRemoteLiveActivities()`.

*SEC-011:* Server must continue binding `deviceId` to authenticated `userId`; clients cannot
register devices for other users.

*SEC-012:* `DELETE /v1/mobile/devices/:deviceId` on sign-out must run on Android. Invalidate FCM
token server-side; clear local registration state.

=== Relay server token storage

[cols="2,3",options="header"]
|===
| Field | Risk

| `relay_mobile_devices.push_token`
| Plaintext FCM/APNs token. Unique index prevents duplicate claims. Cleared on permanent delivery
  failure.

| `relay_delivery_attempts`
| Logs `tokenSuffix` only — preserve for FCM (REL-030).

| Device `label`
| From `Constants.deviceName` — may contain PII (user-named device). Avoid full label in analytics
  (SEC-013).
|===

== Notification payload PII boundaries

=== Allowed in push notification (user-visible)

Derived from `ApnsDeliveries.notificationForAggregate()`:

[cols="2,2,1",options="header"]
|===
| Field | Source | Max length (sanitizer)

| `title` (FCM notification)
| `activity.threadTitle`
| 120 chars (`truncateText`)

| `body` (FCM notification)
| ``${activity.status}: ${activity.projectTitle}``
| 120 chars each component

| `data.environmentId`
| Aggregate row
| Schema-validated `EnvironmentId`

| `data.threadId`
| Aggregate row
| Schema-validated `ThreadId`

| `data.deepLink`
| Aggregate row, sanitized
| 512 chars; must start with `/`; no `//`
|===

=== Explicitly forbidden in push payload

*SEC-030* — relay sanitizer and publisher tests must reject or strip:

* User prompts, model outputs, tool arguments/results
* File paths, repository contents, diff hunks
* Environment credentials, relay tokens, Clerk tokens
* Stack traces with local paths
* Email addresses, API keys, connection strings

*Implementation anchors:* `sanitizeAgentActivityAggregateState`, `sanitizeApnsNotificationPayload`
in `infra/relay/src/agentActivity/agentActivityPayloads.ts`. FCM path must call the same
sanitizers (REL-042).

=== FCM-specific considerations

[cols="2,3",options="header"]
|===
| Topic | Guidance

| Notification vs data-only messages
| Use notification + data for agent alerts (user-visible alert required). Data-only would hide
  from shade but reduce deliverability on OEM kills — product tradeoff documented.

| `android.notification.channel_id`
| Hint only; channels created on device (AGT-002). No sensitive data in channel ID.

| High-priority delivery
| `waiting_for_approval`, `waiting_for_input`, `failed` — visibility on lock screen increases
  exposure of thread *titles* — acceptable per product; document in privacy notice.

| Ongoing notification (M2)
| *SEC-032:* May show phase + thread title in expanded view only; must not stream agent stdout,
  tool output, or file names. Update in place; same PII ceiling as push.

| Google as processor
| FCM message content transits Google infrastructure — treat notification body as
  *confidential-to-Google*, not end-to-end encrypted.
|===

=== Logging and observability boundaries

*SEC-031:* Relay spans may include `relay.environment_id`, `relay.thread_id` — acceptable.
Must not log: full `pushToken`, notification `title`/`body` at info level in production, device
`label` in third-party crash reporters.

== Identified gaps and recommendations

=== P0 (block M1)

. *Split push vs Live Activity gates* in `remoteRegistration.ts` — security-enabling for Android.
. *Shared FCM sanitizer* — extract `notificationForAggregate` + sanitizers for APNs/FCM parity.
. *Android registration tests* — today `remoteRegistration.test.ts` mocks iOS only (qa-strategy
  gap).

=== P1 (block M2)

. *Deep link ACL* (SEC-040) — notification opens thread only after environment authorization check.
. *WebView terminal security profile* — document in `modules/t3-terminal/README.md`; implement
  SEC-050–052 before default-enable.
. *Ongoing notification builder* — separate from push sanitizer; same PII unit tests.

=== P2 (track, not blocking)

. *Plaintext push tokens in DB* — align with org data-classification policy; consider envelope
  encryption at rest for `push_token` column (infra-wide, not Android-only).
. *FCM vs APNs credential blast radius* — custom IAM role with minimal FCM permissions.
. *Device label PII* — optional user-editable label defaulting to generic "Android device".
