== Purpose and scope

This specification defines *Android-native* interaction design for T3 Code Mobile parity with
iOS. T3 Code Mobile is a xref:control-plane[control plane] вҖ” delegate, monitor, approve, review вҖ”
not a pocket IDE.

=== In scope

* Thread chrome (phone + tablet split at 720Г—600)
* Agent phase indicator and notification copy
* Ongoing notification (Android Live Activity substitute, M2)
* Strip-mode composer (M1) and mirror editor handoff (M2)
* Bottom accessory bar (Files | Terminal | Review | Git)
* Terminal tier labeling (Basic / Standard / Native)
* Per-route empty, loading, error, and degraded states
* Accessibility and dark-mode milestone scope

=== Out of scope

* Pixel-identical iOS 26 glass / Liquid Glass
* Kotlin native review diff or composer editor
* Foreground-service UX (M4 optional; see plan В§4.1)
* Home thread list visual redesign (inherits existing patterns)

== Design principles (Android-specific)

These principles apply to all Android parity work. They complement engineering principles in
link:plan.adoc[plan.adoc] В§Architecture strategy.

=== DP-01: Same jobs, different controls

Android does not port iOS Mail `unstable_headerToolbarItems` or glass header toolbars. Thread tools
live in a *bottom accessory bar* on phone and a *left rail* on tablet split. Keyboard shortcuts
(`files`, `terminal`, `review`) map to the same destinations as bar items.

=== DP-02: Material-native chrome, T3 semantics

Use solid sheet backgrounds, elevation for accessory bar, and system back / predictive back. Do not
simulate iOS glass. Phase colors and copy align with iOS Live Activity semantics so relay payloads
stay platform-agnostic.

=== DP-03: Honest capability labeling

When Tier 0 ships (JS review, strip composer, stub terminal), show inline *degraded* affordances вҖ”
never silent failure or crash. Labels: "Basic terminal", "Enhanced terminal", "Native terminal";
composer strip-mode coach mark in M1.

=== DP-04: Agent state is always visible

Interruptive phases (`waiting_for_approval`, `waiting_for_input`, `failed`) surface in thread
chrome, push notifications, and (M2) ongoing notification. Running work uses low-importance channel
+ in-app teal indicator вҖ” not notification spam.

=== DP-05: Thumb-zone primary actions

On phone, approve/respond, send message, and thread tool switching sit in the lower half. Phase
indicator sits above composer or in header sub-row вҖ” never only in status bar.

=== DP-06: Tablet is a first-class layout

At вүҘ720Г—600, persistent sidebar + thread detail + optional inspector pane. Accessory bar becomes
vertical rail; composer and phase indicator stay in the chat pane only (inspector does not steal
thread chrome).

=== DP-07: Notification respect

One ongoing notification per active agent session aggregate; update in place. Dismiss does not
stop agent. Settings must enable Android notification permission flow (replace iOS-only
`unsupported` gate).

=== Update rules

* UX-007: MUST update existing notification (same `notificationTag`); MUST NOT post new ID per tick.
* UX-008: Dismiss swipe removes shade only; agent keeps running.
* UX-009: End ongoing notification when `activeCount === 0` or terminal phase delivered via `agent_completed` / `agent_failed` channel.
* UX-010: Phase color on ongoing notification MUST match <<phase-indicator,phase indicator>> table.

==== Token chip specification

[cols="2,2",options="header"]
|===
| Property | Value

| Height
| *48dp* minimum touch target

| Min width
| 48dp (icon-only): else intrinsic + 16dp horizontal padding

| Corner radius
| 12dp

| Typography
| 14sp `font-t3-medium`; truncate middle with ellipsis

| Icon
| 16dp leading: file icon, skill bolt, or image thumb (24dp in chip)

| Remove affordance
| 24dp `Г—` in trailing gutter (`removeButtonPlacement: "gutter"`); hit slop 8dp

| Spacing
| 8dp gap between chips; 8dp margin below strip

| Max visible before scroll
| 3.5 chips on 360dp width
|===

* UX-011: Token chips MUST be 48dp tall (Material touch target).*

* UX-012: Remove MUST use gutter placement (not overlay) for strip-mode вҖ” distinct from image attachment overlay pattern.*

==== Chip types

[cols="2,2,2",options="header"]
|===
| Token kind | Leading | Remove behavior

| File reference
| Pierre file icon
| Removes token from `collectComposerInlineTokens` payload

| Skill reference
| Skill chip icon
| Same

| Image (M1)
| Thumbnail in strip above field (existing `ComposerAttachmentStrip`)
| Not inline chip in M1
|===

==== Edit behavior (strip-mode)

* Tap chip body (not Г—): move cursor focus to TextInput; no inline edit of chip boundary in M1.
* Long-press chip: TalkBack menu *Remove attachment*.
* Paste `@file` or skill trigger: append token to strip on next parse pass.

* UX-013: Send pipeline MUST preserve token payload identical to iOS (`CMP-001`).*

==== Coach mark (first run)

Files and skills you add show as chips under the text field until mirror editing ships.
[Got it]
____

* Dismiss: persist `strip_composer_coach_seen` in preferences.
* UX-014: Coach mark MUST NOT block send or keyboard.

* Chips render inline in mirror layer; strip hidden.
* Coach mark not shown.
* Degradation banner removed.

Transition: no migration UI; mirror enabled via capability on OTA.
