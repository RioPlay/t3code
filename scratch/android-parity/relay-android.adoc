== Problem statement

T3 Code Relay agent-awareness push delivery is *iOS-only end-to-end*. An Android user who signs in,
links an environment, and grants notification permission never receives agent alerts because the
stack rejects or skips Android at every layer.

=== Current constraints (as of 2026-07-06)

[cols="2,3",options="header"]
|===
| Layer | iOS-only behavior

| Contracts (`packages/contracts/src/relay.ts`)
| `RelayAgentAwarenessPlatform = Schema.Literal("ios")`; `iosMajorVersion >= 18` required on all
  registration payloads and device records.

| Mobile registration (`apps/mobile/src/features/agent-awareness/`)
| `registrationPayload.ts` hardcodes `platform: "ios"`. `remoteRegistration.ts` gates all
  registration on `canRegisterRemoteLiveActivities()` (`Platform.OS === "ios"`), accepts only
  `token.type === "ios"` from `Notifications.getDevicePushTokenAsync()`.

| Relay persistence (`infra/relay/src/persistence/schema.ts`)
| `relay_mobile_devices.platform` typed `"ios"` only; `ios_major_version` `NOT NULL`.

| Relay delivery (`infra/relay/src/agentActivity/ApnsDeliveries.ts`)
| All `push_notification` and Live Activity deliveries go through APNs HTTP/2. No FCM path.

| Publisher routing (`infra/relay/src/agentActivity/AgentActivityPublisher.ts`)
| Invokes `ApnsDeliveries.sendForTarget` for every registered device regardless of platform
  (would fail silently or mis-deliver if Android tokens were stored).

| Web client UI (`apps/web/.../MobileClientsUserProfilePage.logic.ts`)
| Device list labels assume iOS (`iOS ${iosMajorVersion}`).
|===

=== User-visible impact

* Android users cannot register a push token with relay → no approval/input/failure/completion alerts.
* Android is excluded from the agent *control plane* (see link:plan.adoc[plan.adoc] §Product north stars).
* iOS Live Activities remain iOS-only; Android substitute (ongoing notification, M2) depends on this
  workstream delivering reliable FCM `push_notification` delivery first.

=== Scope of this workstream

[cols="1,1",options="header"]
|===
| In scope | Out of scope

| Contract + DB schema for Android device registration
| Android Live Activity / Glance widgets (iOS `expo-widgets` only)

| FCM push delivery mirroring APNs `push_notification` path
| Ongoing notification UX on device (mobile M2; consumes FCM)

| Relay publisher platform routing
| Managed tunnels, environment link, DPoP (unchanged)

| Firebase + EAS credential wiring for mobile builds
| Web profile page full redesign (minimal Android label only)

| Staging E2E: publish activity → FCM → tap → deep link
| Foreground service (M4 optional)
|===

== Target architecture

[source]
----
Environment publishes agent activity
        │
        ▼
AgentActivityPublisher.publishForDeliveryUser
        │
        ├── listTargets (join relay_mobile_devices + relay_live_activities)
        │
        └── per target ──► platform dispatch
                 │
     ┌───────────┴───────────┐
     ▼                       ▼
platform === "ios"     platform === "android"
     │                       │
ApnsDeliveries          FcmDeliveries
(live_activity_*       (push_notification
 + push_notification)   only)
     │                       │
APNs HTTP/2              FCM HTTP v1
     │                       │
iOS app                  Android app (expo-notifications)
----

*Design principle:* Reuse the existing `pushToken` column and `push_notification` delivery kind for
FCM registration tokens. Do *not* introduce a parallel `fcmToken` field in the HTTP API — the
platform discriminator tells relay how to interpret `pushToken`.

== Schema and contract changes

=== Platform literal

[source,typescript]
----
// packages/contracts/src/relay.ts
export const RelayAgentAwarenessPlatform = Schema.Literals(["ios", "android"]);
----

=== Registration request — discriminated union (recommended)

Replace the flat `RelayDeviceRegistrationRequest` struct with a tagged union so invalid
platform/field combinations are rejected at decode time.

[source,typescript]
----
const RelayDeviceRegistrationBase = {
  deviceId: TrimmedNonEmptyString,
  label: TrimmedNonEmptyString,
  appVersion: Schema.optional(TrimmedNonEmptyString),
  pushToken: Schema.optional(TrimmedNonEmptyString),
  preferences: RelayAgentAwarenessPreferences,
} as const;

export const RelayIosDeviceRegistrationRequest = Schema.Struct({
  ...RelayDeviceRegistrationBase,
  platform: Schema.Literal("ios"),
  iosMajorVersion: Schema.Int.check(Schema.isGreaterThanOrEqualTo(18)),
  pushToStartToken: Schema.optional(TrimmedNonEmptyString),
});

export const RelayAndroidDeviceRegistrationRequest = Schema.Struct({
  ...RelayDeviceRegistrationBase,
  platform: Schema.Literal("android"),
  androidSdkVersion: Schema.Int.check(Schema.isGreaterThanOrEqualTo(26)),
  // pushToStartToken MUST NOT appear — decode fails if present (use Struct omit or filter)
});

export const RelayDeviceRegistrationRequest = Schema.Union([
  RelayIosDeviceRegistrationRequest,
  RelayAndroidDeviceRegistrationRequest,
]);
----

[cols="2,2,2",options="header"]
|===
| Field | iOS | Android

| `platform`
| `"ios"`
| `"android"`

| `iosMajorVersion`
| Required, `>= 18`
| *Absent* (must not be sent)

| `androidSdkVersion`
| *Absent*
| Required, `>= 26` (Android 8.0 — notification channels baseline)

| `pushToken`
| APNs device token (hex)
| FCM registration token from `getDevicePushTokenAsync()` when `type === "android"`

| `pushToStartToken`
| Optional (Live Activity remote start)
| *Must not be sent* — relay ignores if present; prefer schema rejection

| `preferences.liveActivitiesEnabled`
| User preference
| Always `false` on mobile client; relay may coerce on ingest

| `preferences.notificationsEnabled`
| User preference
| User preference (same semantics)
|===

==== Android `pushToken` semantics (REL-010)

* Mobile obtains token via `Notifications.getDevicePushTokenAsync()` after permission grant.
* When `token.type === "android"`, send `token.data` as `pushToken` (trimmed non-empty string).
* Same uniqueness rules as iOS: relay claims token globally via
  `idx_relay_mobile_devices_push_token` before upsert (`Devices.register`).

==== Minimum SDK rationale

`androidSdkVersion >= 26` matches notification channel support required for AGT-002 channel
strategy in link:requirements.adoc[requirements.adoc]. Relay stores the value for support/debug;
it does not gate delivery server-side beyond schema validation.

=== Delivery result contract extension

`RelayDeliveryResult` today exposes `apnsStatus`, `apnsReason`, `apnsId`. Add parallel optional
FCM fields *without* breaking existing iOS clients:

[source,typescript]
----
export const RelayDeliveryResult = Schema.Struct({
  deviceId: TrimmedNonEmptyString,
  kind: RelayDeliveryKind,
  ok: Schema.Boolean,
  queued: Schema.optional(Schema.Boolean),
  apnsStatus: Schema.NullOr(Schema.Number),
  apnsReason: Schema.NullOr(Schema.String),
  apnsId: Schema.NullOr(Schema.String),
  fcmStatus: Schema.optional(Schema.NullOr(Schema.String)),   // e.g. "NOT_FOUND", "UNAVAILABLE"
  fcmReason: Schema.optional(Schema.NullOr(Schema.String)),
  fcmMessageId: Schema.optional(Schema.NullOr(Schema.String)),
});
----

iOS deliveries continue populating `apns*` fields; Android populates `fcm*`. OpenAPI consumers
treat absent `fcm*` as N/A.

=== Live Activity endpoints

`POST /v1/mobile/live-activities` remains *iOS-only*. Relay returns `400` with a clear error if
the device row has `platform = android` (REL-022). Mobile Android client must not call this
endpoint (already gated by `canRegisterRemoteLiveActivities()`).

==== FcmDeliveries behavior

* *Only* handles `kind === "push_notification"`.
* `sendForTarget` for Android:
** Skip all `live_activity_*` kinds (return `null`).
** Reuse `notificationForAggregate()` logic from `ApnsDeliveries.ts` (extract shared
   `choosePushNotificationForTarget` helper).
* Permanent token failures (FCM `NOT_FOUND`, `UNREGISTERED`) → clear `push_token` on device row
  (mirror `invalidateDeliveryToken` for `push_notification`).
* Delivery attempts: record `fcmStatus` / `fcmReason` / `fcmMessageId` in extended
  `relay_delivery_attempts` columns (add via migration) OR map into existing `transport_error`
  until column migration lands — prefer explicit columns for ops.

==== FCM HTTP v1 message shape

Align custom data with APNs custom payload so mobile `notificationPayload.ts` works unchanged:

[source,json]
----
{
  "message": {
    "token": "<fcm-registration-token>",
    "notification": {
      "title": "<threadTitle>",
      "body": "<status>: <projectTitle>"
    },
    "data": {
      "environmentId": "<id>",
      "threadId": "<id>",
      "deepLink": "/threads/<env>/<thread>"
    },
    "android": {
      "priority": "HIGH",
      "notification": {
        "channel_id": "agent_approval"
      }
    }
  }
}
----

Channel ID in FCM payload is a *hint*; mobile client creates channels locally (AGT-002). Map
phase → channel:

[cols="2,2",options="header"]
|===
| Phase | FCM `android.notification.channel_id`

| `waiting_for_approval`
| `agent_approval`

| `waiting_for_input`
| `agent_input`

| `failed`
| `agent_failed`

| `completed`
| `agent_completed`

| `running`, `starting`, other
| `agent_running`
|===

Extend `infra/relay/src/Config.ts`:

export interface FcmCredentials {

  readonly privateKey: Redacted.Redacted<string>;

// RelayConfiguration service adds:

Worker env vars (`infra/relay/src/worker.ts`):

* `FCM_PROJECT_ID`
* `FCM_CLIENT_EMAIL`
* `FCM_PRIVATE_KEY` (redacted)
* `FCM_DELIVERY_JOB_SIGNING_SECRET` (auto-generated like APNs)
* New queue binding: `RelayFcmDeliveryQueue` (+ DLQ)

Runtime layer adds `FcmClient`, `FcmDeliveries`, `FcmDeliveryQueue` alongside APNs equivalents.

=== Publisher routing change

`AgentActivityPublisher.publishForDeliveryUser` replaces direct `ApnsDeliveries` calls with
`MobileDeliveries.sendForTarget` / `sendPushNotificationForTarget`:

[source,typescript]
----
// Pseudocode
function deliveryLayerFor(target: TargetRow) {
  return target.platform === "android" ? fcmDeliveries : apnsDeliveries;
}
----

`replayForLiveActivityRegistration` on Android: only replay push notification if notifications
enabled and aggregate warrants it — skip Live Activity replay entirely.

=== Devices service

`Devices.register` must branch on `registration.platform`:

* Persist `iosMajorVersion` or `androidSdkVersion`, not both.
* For Android: force `preferences.liveActivitiesEnabled = false` in stored JSON (defense in depth).
* Do not write `pushToStartToken` for Android registrations.

relay. Restrict to FCM send scope in production IAM where possible.

== Migration strategy for existing iOS devices

=== Data plane (zero-downtime)

[cols="1,3",options="header"]
|===
| Step | Action

| 1
| Deploy DB migration: nullable `ios_major_version`, add `android_sdk_version`. *No row updates
  required* — existing rows keep `platform = 'ios'` and non-null `ios_major_version`.

| 2
| Deploy relay with backward-compatible contract decode: still accepts existing iOS payloads.

| 3
| Deploy relay with FCM modules *disabled* until credentials configured (feature flag
  `RELAY_FCM_DELIVERY_ENABLED`).

| 4
| Configure FCM credentials in staging; enable flag; validate Android E2E.

| 5
| Production: enable FCM flag after staging sign-off. iOS traffic unchanged on APNs path.
|===

=== API compatibility

* Existing iOS mobile builds (pre-Android contract) continue sending flat struct with
  `platform: "ios"` — supported via union member `RelayIosDeviceRegistrationRequest`.
* `GET /v1/client/devices` returns discriminated records; web UI must handle `platform: "android"`
  (minimal label change).
* OpenAPI version bump: minor (additive enum value + union members).

=== Token namespace

iOS APNs tokens and Android FCM tokens are both stored in `push_token`. The unique index
prevents the same string from attaching to two devices — safe because token formats differ.
Cross-platform token collision is negligible.

=== Rollback

Disable `RELAY_FCM_DELIVERY_ENABLED` → Android registrations still accepted (for forward
compat) but delivery skipped with logged warning. iOS unaffected.

== Rollout and feature flags

[cols="2,2,2",options="header"]
|===
| Flag / gate | Default | Effect

| `RELAY_FCM_DELIVERY_ENABLED`
| `false` in prod until E2E sign-off
| Master switch for FCM send path

| `RELAY_ANDROID_REGISTRATION_ENABLED`
| `true` when contract deployed
| Reject `platform: android` with `503` when false (optional kill switch)

| Mobile: no flag required
| —
| Android registration ships with mobile M1 agent-awareness work

| Staging relay credentials
| FCM service account configured
| Required before enabling `RELAY_FCM_DELIVERY_ENABLED`
|===

=== Rollout sequence

. Merge contract + DB migration (registration accepted, delivery may no-op).
. Deploy relay with FCM code behind `RELAY_FCM_DELIVERY_ENABLED=false`.
. Configure staging FCM secrets; enable flag on staging.
. Complete staging E2E checklist.
. Ship Android dev/preview builds with `google-services.json`.
. Enable production FCM flag; monitor `relay_delivery_attempts` error rate + FCM quotas.

== Security considerations

=== Push token storage

* Tokens stored in plaintext `relay_mobile_devices.push_token` (existing iOS behavior).
  APNs and FCM tokens are credentials — treat DB access as sensitive.
* Unique index on `push_token` prevents token replay across users/devices.
* On permanent delivery failure, token nulled via `invalidateDeliveryToken` (limit exposure window).
* Delivery attempt logs store only `tokenSuffix` (last 8 chars) — preserve this pattern for FCM.

=== PII and payload content

[cols="2,3",options="header"]
|===
| Field | Classification

| `userId`, `deviceId`
| Account identifiers — present in DB and traces; follow relay trace redaction policies.

| `label` (device name)
| May contain user-assigned device name from `Constants.deviceName` — avoid logging in full in
  third-party analytics.

| Notification `title` / `body`
| Thread title and status string only — *no* user prompts, file paths, or model outputs.
  Enforced by `notificationForAggregate()` using aggregate row fields sanitized by
  `sanitizeAgentActivityAggregateState`.

| `deepLink`
| Internal path only; `sanitizeDeepLink` blocks `//` and off-root paths.

| FCM / APNs credentials
| Worker secrets via `Redacted`; rotate on compromise; separate dev/staging/prod service accounts.
|===

=== Threat model notes

* *Forged registration:* Mitigated by DPoP-bound `mobile:registration` scope (unchanged).
* *Cross-user token binding:* Mitigated by push token claim-before-upsert in `Devices.register`.
* *Relay send abuse:* Environment publish requires environment credential JWT (unchanged).
* *FCM key leakage:* Service account JSON grants broader than APNs key — store in secrets
  manager, audit access, use least-privilege Firebase IAM role custom role with
  `cloudmessaging.messages.create` only where supported.

=== Compliance

* Android notification permission is runtime opt-in — relay must respect `notificationsEnabled:
  false` in preferences (no "shadow" sends).
* Users can unregister via `DELETE /v1/mobile/devices/:deviceId` — ensure Android sign-out path
  calls this (existing mobile behavior).
